    TP2 — Correction et durcissement d'une machine Linux vulnérable

    Contexte

    Ce TP fait suite au TP1 — Audit guidé d'une machine Linux vulnérable.

    Dans le TP1, vous avez identifié plusieurs vulnérabilités visibles à l'aide des mécanismes Linux de base :

    • utilisateurs ;
    • groupes ;
    • fichiers sensibles ;
    • permissions ;
    • sudo ;
    • services ;
    • principe du moindre privilège.

    Dans ce TP2, vous devez maintenant corriger les vulnérabilités identifiées et compléter l'analyse avec des mécanismes de sécurité plus avancés :

    • ACL ;
    • Linux Capabilities ;
    • MAC avec AppArmor ;
    • confinement d'application.

    Vous travaillez toujours sur la machine de l'entreprise fictive NovaTech.

    Objectifs

    À l'issue de ce TP, vous serez capable de :

    • Corriger des permissions dangereuses ;
    • Réduire les privilèges excessifs ;
    • Corriger une configuration sudo trop permissive ;
    • Utiliser les ACL pour analyser des droits d'accès avancés ;
    • Identifier et supprimer des capabilities dangereuses ;
    • Vérifier l'état d'un mécanisme MAC ;
    • Activer ou renforcer le confinement d'une application ;
    • Vérifier qu'une correction est réellement effective ;
    • Produire une synthèse avant/après.

    Important

    Contrairement au TP1, vous devez maintenant corriger les vulnérabilités.

    Chaque correction doit être justifiée et vérifiée.

    Pour chaque vulnérabilité corrigée, votre compte rendu devra contenir :

    • la vulnérabilité initiale ;
    • la commande de correction ;
    • la commande de vérification ;
    • le résultat obtenu ;
    • une courte explication de l'amélioration de sécurité.

    Dans le TP1, vous disposiez uniquement d'un accès au compte student afin de réaliser un audit sans modifier le système.

    Dans le TP2, votre rôle est celui d'un administrateur chargé de corriger les vulnérabilités identifiées.

    Vous devez donc utiliser le compte root avec le mot de passe change_me_securely.

    Compte rendu

    Un compte rendu individuel est attendu à l'issue du TP.

    Celui-ci devra inclure :

    • les vulnérabilités corrigées ;
    • les commandes utilisées ;
    • les preuves avant/après ;
    • les vérifications réalisées ;
    • les risques résiduels éventuels.

    Le compte rendu devra être transmis à la fin de la séance.

    Dans le TP1, vous avez réalisé un audit guidé de la machine Linux vulnérable.

    Dans ce TP2, vous disposez directement de la liste des vulnérabilités confirmées par l'audit.

    Votre mission consiste maintenant à :

    • corriger chaque vulnérabilité obligatoire ;
    • justifier la correction appliquée ;
    • vérifier que la correction est effective ;
    • expliquer l'amélioration de sécurité obtenue.

    Important

    Vous devez corriger les vulnérabilités dans l'ordre de priorité indiqué.

    Les vulnérabilités VULN-01 à VULN-10 sont obligatoires.

    Les vulnérabilités VULN-11 à VULN-15 sont des vulnérabilités avancées ou bonus. Elles permettent d'approfondir les notions d'ACL, de capabilities, de MAC et de confinement.

    Exercice 2 : Corrections de base

    Contexte

    Cette première étape consiste à corriger les vulnérabilités visibles avec les mécanismes Linux classiques.

    Vous devez appliquer le principe du moindre privilège :

    Un utilisateur, un fichier ou une application ne doit disposer que des droits strictement nécessaires.

    Question 1

    Corrigez les problèmes liés aux utilisateurs ou groupes.

    Vous devez traiter au minimum :

    VULN-02

    Correction de sudo

    Question 3

    Corrigez la règle sudo trop permissive.

    Exemple de règle dangereuse :

    alice ALL=(ALL) NOPASSWD:ALL

    Cette règle permet à alice d'exécuter n'importe quelle commande avec les privilèges administrateur, sans authentification supplémentaire.

    Complétez le tableau :

    ID Règle dangereuse Risque Correction proposée Vérification
    VULN-07
    💡 Indices

    Commandes utiles :

    sudo visudo

sudo visudo -f /etc/sudoers.d/novatech

sudo -l -U alice
sudo -l

    Exemple de règle plus limitée

    Une règle plus conforme au principe du moindre privilège pourrait autoriser uniquement une action précise :

    alice ALL=(root) /usr/bin/systemctl restart webapp

    Cette règle est beaucoup moins dangereuse que :

    alice ALL=(ALL) NOPASSWD:ALL

    Exercice 3 : ACL — droits d'accès avancés

    Contexte

    Les permissions classiques Linux reposent sur trois catégories :

    • propriétaire ;
    • groupe ;
    • autres.

    Cependant, il existe un mécanisme plus fin : les ACL.

    Les ACL permettent d'accorder des droits spécifiques à un utilisateur ou à un groupe, même si ces droits ne sont pas visibles directement avec une simple lecture des permissions classiques.

    Découverte des ACL

    Question 1

    Analysez les ACL du répertoire projet.

    Vous devez traiter :

    • VULN-10 : ACL excessive sur /srv/project.

    Complétez le tableau :

    ID Ressource ACL observée Utilisateur/groupe concerné Droit accordé Risque
    VULN-10 /srv/project
    💡 Indices

    Commandes utiles :

    getfacl /srv/project
getfacl /srv/project/*

    Question 2

    Comparez les informations obtenues avec :

    ls -l

    et avec :

    getfacl

    Répondez aux questions suivantes :

    • quelles informations supplémentaires sont visibles avec getfacl ?
    • pourquoi ls -l peut-il être insuffisant pour auditer les droits ?
    • quel utilisateur dispose d'un accès trop large ?

    Correction des ACL dangereuses

    Question 3

    Supprimez l'ACL excessive.

    Exemple : si l'utilisateur intern dispose d'un droit d'écriture injustifié sur /srv/project, supprimez ce droit.

    Exercice 4 : Linux Capabilities

    Contexte

    Sous Linux, le compte root possède tous les privilèges.

    Les Linux Capabilities permettent de découper certains privilèges de root en permissions plus petites.

    Cela permet à un programme de disposer uniquement du privilège dont il a besoin, sans être exécuté entièrement en tant que root.

    Cependant, une capability mal attribuée peut devenir dangereuse.

    Identification des capabilities

    Question 1 — Bonus

    Vous pouvez traiter :

    Complétez le tableau :

    ID Fichier Capability Justification apparente Risque
    VULN-12
    VULN-13
    💡 Indices

    Commandes utiles :

    Ne supprimez pas au hasard toutes les capabilities du système.

    Votre objectif est d'identifier et corriger celles qui sont injustifiées ou dangereuses.

    Exercice 5 : MAC et confinement d'application

    Contexte

    Les permissions classiques et les ACL relèvent principalement du contrôle d'accès discrétionnaire.

    Avec le contrôle d'accès obligatoire, ou MAC pour Mandatory Access Control, les accès sont limités par une politique de sécurité.

    Une application compromise ne doit pas pouvoir accéder librement à tout le système.

    Dans ce TP, nous utilisons AppArmor comme exemple de mécanisme MAC.

    État d'AppArmor

    Activation du confinement

    Question 3 — Bonus

    Recherchez si l'application Web NovaTech dispose d'un profil AppArmor.

    Si un profil existe mais n'est pas en mode enforce, activez le mode enforce.

    Vous pouvez traiter :

    • VULN-15 : application Web insuffisamment confinée.

    Complétez le tableau :

    ID Application Profil AppArmor Mode initial Commande appliquée Mode final
    VULN-15 NovaTech WebApp
    💡 Indices

    Commandes utiles :

    sudo aa-status

sudo aa-status

    Attention

    Si le nom exact du profil est différent, adaptez la commande.

    Vous pouvez lister les profils avec :

    sudo aa-status
ls /etc/apparmor.d/

    Question 4 — Bonus

    Expliquez comment AppArmor limite l'impact d'une compromission de l'application Web.

    À retenir

    • Une correction doit toujours être vérifiée.
    • Les permissions classiques ne montrent pas toujours tous les droits réels.
    • Les ACL permettent un contrôle plus fin, mais doivent être auditées.
    • Les capabilities réduisent le besoin d'exécuter un programme en root, mais peuvent être dangereuses si elles sont mal attribuées.
    • Le MAC, avec AppArmor ou SELinux, ajoute une politique de sécurité indépendante des permissions classiques.
    • Le confinement limite l'impact d'une application compromise.
    • Le durcissement d'un système repose sur la défense en profondeur.

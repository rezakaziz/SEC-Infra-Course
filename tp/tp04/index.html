    Contrôle d'accès sous Linux : ACL, SELinux et AppArmor

    ⬇️ Télécharger la machine virtuelle (vm-sec-tp4.ova)

    Contexte

    Une entreprise héberge plusieurs applications sur un serveur Linux.

    Les administrateurs souhaitent renforcer la protection des données sensibles en limitant les accès aux seuls utilisateurs autorisés.

    Malgré les permissions Linux classiques, certains besoins métiers nécessitent un contrôle plus fin des accès et un confinement des applications.

    Votre mission consiste à mettre en œuvre différents mécanismes de contrôle d'accès afin de protéger les ressources du système.

    Exercice 1 : Analyse des permissions Linux

    Avant de mettre en place des mécanismes avancés, il est nécessaire d'analyser les permissions existantes.

    Question 1

    Identifiez les utilisateurs présents sur la machine :

    cat /etc/passwd

    Question 2

    Examinez les permissions du répertoire suivant :

    ls -ld /srv/rh

    Répondez aux questions suivantes :

    • Qui est propriétaire du répertoire ?
    • Quels utilisateurs peuvent y accéder ?
    • Les permissions actuelles sont-elles suffisantes ?

    Question 3

    Connectez-vous avec différents utilisateurs et vérifiez les accès autorisés ou refusés.

    Exercice 2 : Mise en œuvre des ACL

    L'entreprise souhaite autoriser un utilisateur externe au service RH à consulter certains documents sans modifier les permissions globales.

    Question 1

    Affichez les ACL actuellement configurées :

    getfacl /srv/rh

    Question 2

    Accordez à l'utilisateur auditeur un accès en lecture sur le répertoire RH :

    setfacl

    Question 3

    Vérifiez les nouvelles ACL :

    getfacl /srv/rh

    Question 4

    Testez les accès avec :

    • un utilisateur RH ;
    • l'utilisateur auditeur ;
    • un utilisateur standard.

    Les résultats sont-ils conformes aux attentes ?

    Exercice 3 : Introduction à SELinux

    Le serveur utilise SELinux afin de limiter les actions autorisées aux services système.

    Question 1

    Vérifiez l'état de SELinux :

    getenforce

    Quel mode est actuellement utilisé ?

    Question 2

    Affichez les contextes de sécurité du répertoire Web :

    ls -Z /var/www/html

    Que remarquez-vous ?

    Question 3

    Consultez les contextes associés aux processus Apache :

    ps -eZ | grep httpd

    Quel est l'intérêt de cette séparation ?

    Exercice 4 : Analyse d'un refus SELinux

    Un administrateur a déplacé un fichier Web vers un nouveau répertoire.

    Le site ne fonctionne plus correctement.

    Question 1

    Identifiez les erreurs générées :

    sudo ausearch -m avc

    ou

    sudo journalctl | grep denied

    Question 2

    Déterminez la cause du problème.

    Question 3

    Corrigez le contexte de sécurité approprié.

    Vérifiez ensuite que le service fonctionne correctement.

    TP1 — Audit guidé d'une machine Linux vulnérable

    ⬇️ Télécharger la machine virtuelle (sec-tp-vm.ova)

    Contexte et organisation du TP

    Vous intervenez en tant qu'auditeur sécurité junior pour l'entreprise fictive NovaTech.

    Vous disposez d'un accès au compte student et devez réaliser un audit de la machine afin d'identifier des vulnérabilités liées aux utilisateurs, groupes, permissions, fichiers sensibles, règles sudo, services et au principe du moindre privilège.

    Ce TP est volontairement plus riche que ce qui peut être réalisé intégralement pendant la durée de la séance.

    L'objectif n'est pas nécessairement de terminer l'ensemble du TP pendant la séance.

    L'objectif principal est de vous familiariser avec les méthodes d'audit et les mécanismes de sécurité étudiés en cours. Vous êtes libres de poursuivre et terminer le TP en autonomie après la séance si vous le souhaitez.

    Compte rendu

    À la fin de la séance, vous devez transmettre un compte rendu intermédiaire individuel contenant au minimum :

    • les réponses aux exercices 1 et 2 ;
    • les commandes utilisées ;
    • les observations réalisées ;
    • les vulnérabilités identifiées ;
    • les preuves associées.

    Les exercices suivants pourront être complétés ultérieurement et intégrés dans une version finale du compte rendu.

    Adresse de destination :

    rezak.aziz@lecnam.net

    Objet du mail :

    [TP1 U Paris Sécurité] Nom Prénom

    • PDF

    Exercice 2 : Utilisateurs, groupes et comptes suspects

    Contexte

    Linux est un système multi-utilisateurs.

    Une mauvaise gestion des comptes ou des groupes peut permettre à un utilisateur d'accéder à des ressources auxquelles il ne devrait pas avoir accès.

    Votre objectif est d'identifier les utilisateurs présents sur la machine et de repérer les comptes ou appartenances de groupe suspects.

    Utilisateurs du système

    Question 1

    Identifiez les utilisateurs disposant d'un répertoire personnel dans /home.

    Complétez le tableau suivant :

    Utilisateur UID GID Répertoire personnel Shell

    Question 2

    Afficher les autres utilisateurs et classez les.

    Utilisateur Type supposé Justification
    Humain / Système / Suspect

    Rappel

    Un utilisateur humain possède souvent :

    • un répertoire personnel dans /home ;
    • un shell interactif comme /bin/bash ou /bin/sh ;
    • un UID généralement supérieur ou égal à 1000 selon les distributions.

    Un utilisateur système est souvent utilisé pour exécuter un service.

    Exemples de situations possibles :

    • compte inutilisé ;

    • compte de test encore présent ;

    Exercice 3 : Fichiers sensibles et permissions dangereuses

    Contexte

    Une machine Linux contient de nombreuses ressources sensibles :

    • fichiers utilisateurs ;
    • clés SSH ;
    • fichiers de configuration ;
    • mots de passe ou secrets ;
    • scripts d'administration ;
    • fichiers de log.

    Une mauvaise protection de ces fichiers peut entraîner une fuite d'information ou une modification non autorisée.

    Répertoires sensibles

    Question 1

    Analysez rapidement les répertoires suivants :

    /home
/srv/project
/srv/webapp
/opt/backup
/var/log
/tmp

    Complétez le tableau :

    Répertoire Rôle supposé Propriétaire Groupe Permissions Risque si mal protégé

    Fichiers sensibles

    Question 2

    Recherchez des fichiers sensibles dans /home, /srv et /opt.

    Vous devez identifier au minimum :

    • une clé SSH ;
    • un fichier de configuration ;
    • un fichier contenant un mot de passe, secret ou token ;
    • un script d'administration ou de sauvegarde.

    Complétez le tableau :

    Fichier Chemin Propriétaire Permissions Pourquoi est-ce sensible ?

    Permissions trop ouvertes

    Question 3

    Recherchez les fichiers :

    • lisibles par tous les utilisateurs ;
    • modifiables par tous les utilisateurs.

    Complétez le tableau :

    Fichier Permissions Problème observé Risque

    Question 4

    Analysez particulièrement le fichier suivant :

    /opt/backup/backup.sh

    Indiquez :

    • son propriétaire ;
    • son groupe ;
    • ses permissions ;
    • si le compte student peut le lire ;
    • si le compte student peut le modifier ;
    • pourquoi cela peut être dangereux.

    Question 5

    Expliquez pourquoi les situations suivantes sont dangereuses :

    • une clé privée SSH lisible par d'autres utilisateurs ;
    • un fichier de configuration contenant un mot de passe ;
    • un script de sauvegarde modifiable par tous ;
    • un fichier sensible avec des permissions 777.
    💡 Indices

    Commandes utiles :

    ls -ld /home

ls -ld /srv/webapp
ls -ld /opt/backup

ls -ld /tmp

find /home /srv /opt -type f -perm -o+r 2>/dev/null
find /home /srv /opt -type f -perm -o+w 2>/dev/null

grep -Ri "password" /home /srv /opt 2>/dev/null
grep -Ri "secret" /home /srv /opt 2>/dev/null
grep -Ri "token" /home /srv /opt 2>/dev/null
grep -Ri "key" /home /srv /opt 2>/dev/null

ls -l /opt/backup/backup.sh
cat /opt/backup/backup.sh

    Exercice 4 : Sudo, services et surface d'attaque

    Contexte

    Le compte root possède tous les privilèges sur le système.

    La commande sudo permet à certains utilisateurs d'exécuter des commandes avec des privilèges élevés.

    Un service inutile ou exécuté avec trop de privilèges augmente la surface d'attaque.

    Analyse de sudo

    Question 1

    Vérifiez les privilèges sudo du compte student.

    Complétez le tableau :

    Utilisateur Commandes sudo autorisées Mot de passe requis ? Risque

    Question 2

    Recherchez si une règle sudo semble trop permissive.

    Vous devez notamment vérifier :

    • l'utilisation de NOPASSWD ;
    • la possibilité d'exécuter toutes les commandes ;
    • la possibilité d'exécuter un script sensible avec des privilèges élevés.

    Complétez le tableau :

    Utilisateur Règle sudo observée Pourquoi est-ce dangereux ? Recommandation

    Question 3

    Expliquez pourquoi la règle suivante est dangereuse :

    alice ALL=(ALL) NOPASSWD:ALL

    Proposez une règle plus conforme au principe du moindre privilège.

    Services actifs

    Question 4

    Identifiez les services réseau en écoute sur la machine.

    Complétez le tableau :

    Port Protocole Service supposé Utilisateur / processus Nécessaire ?

    Question 5

    Recherchez si une application ou un service s'exécute avec des privilèges trop élevés.

    Répondez aux questions suivantes :

    • l'application est-elle exécutée par root ?
    • devrait-elle réellement être exécutée par root ?
    • quel serait le risque si cette application était compromise ?
    • quel type de compte devrait être utilisé à la place ?
    💡 Indices

    Commandes utiles :

    sudo -l

ls -l /etc/sudoers
ls -l /etc/sudoers.d/
sudo cat /etc/sudoers.d/*

ss -tulpen
ss -tulpn
ps aux
ps aux | grep webapp
systemctl --type=service --state=running

    Recommandations générales

    Question 3

    Formulez au moins cinq recommandations de sécurité pour améliorer la machine.

    Vos recommandations doivent porter sur au moins quatre des thèmes suivants :

    • comptes utilisateurs ;
    • groupes ;
    • permissions ;
    • fichiers sensibles ;
    • sudo ;
    • services ;
    • principe du moindre privilège.

    À retenir

    • Un audit commence par l'identification de l'environnement.
    • Les utilisateurs et groupes déterminent une grande partie des droits d'accès.
    • Les fichiers sensibles doivent être identifiés et protégés.
    • Les permissions trop ouvertes exposent la confidentialité et l'intégrité.
    • Une mauvaise configuration de sudo peut conduire à une compromission complète.
    • Les services inutiles augmentent la surface d'attaque.
    • Le principe du moindre privilège doit guider toutes les recommandations.

    Séance 6 – Contrôle d'accès obligatoire et confinement des applications

    Objectifs de la séance

    À l'issue de cette séance, l'étudiant sera capable de :

    • Comprendre les limites du contrôle d'accès discrétionnaire (DAC) ;
    • Comprendre le principe du contrôle d'accès obligatoire (MAC) ;
    • Identifier le rôle de SELinux et d'AppArmor ;
    • Comprendre le principe du confinement des applications ;
    • Expliquer comment le confinement participe à la défense en profondeur.

    1. Introduction

    Retour sur la séance précédente

    Nous avons étudié :

    • les permissions Unix ;
    • les utilisateurs et groupes ;
    • les ACL.

    Ces mécanismes permettent de contrôler l'accès aux ressources.

    Question :

    Que se passe-t-il lorsqu'une application autorisée est compromise ?

    Exemple

    Un serveur Web Apache dispose d'un accès à certains fichiers.

    Si l'application est compromise :

    • elle conserve ses droits ;
    • l'attaquant hérite de ces droits.

    Les permissions classiques ne suffisent donc pas toujours.

    2. Limites du modèle DAC

    Contrôle d'accès discrétionnaire (DAC)

    Le propriétaire décide :

    • qui peut accéder ;
    • quelles actions sont autorisées.

    Limites

    Erreur humaine

    Permissions trop permissives.

    Application compromise

    L'application conserve ses droits légitimes.

    Propagation de l'attaque

    Une compromission peut entraîner :

    • accès aux données ;
    • modification de fichiers ;
    • exécution de commandes.

    3. Contrôle d'accès obligatoire (MAC)

    Principe

    Le système applique une politique de sécurité indépendante de l'utilisateur.

    L'accès dépend :

    • de règles de sécurité ;
    • du contexte d'exécution ;
    • de la politique définie par l'administrateur.

    Objectifs

    • limiter les conséquences d'une compromission ;
    • isoler les services ;
    • protéger les ressources sensibles.

    Comparaison DAC / MAC

    DAC MAC
    Basé sur les permissions Basé sur une politique de sécurité
    Contrôlé par le propriétaire Contrôlé par le système
    Flexible Plus restrictif
    Protection des ressources Protection des ressources et des applications

    4. SELinux

    Présentation

    SELinux (Security Enhanced Linux) est un mécanisme MAC développé initialement par la NSA.

    Principalement utilisé sur :

    • Red Hat Enterprise Linux ;
    • Rocky Linux ;
    • AlmaLinux ;
    • Fedora.

    Principe général

    Chaque ressource possède un contexte de sécurité.

    Exemples :

    • fichiers ;
    • processus ;
    • ports réseau.

    Modes de fonctionnement

    Enforcing

    La politique est appliquée.

    Permissive

    Les violations sont enregistrées mais non bloquantes.

    Disabled

    SELinux est désactivé.

    Intérêt

    Limiter les actions d'un service même lorsqu'il est compromis.

    5. AppArmor

    Présentation

    AppArmor est le mécanisme MAC utilisé par défaut sous Ubuntu et Debian.

    Principe

    Les applications sont associées à des profils décrivant :

    • les fichiers accessibles ;
    • les opérations autorisées ;
    • les ressources utilisables.

    Modes de fonctionnement

    Enforce

    Les règles sont appliquées.

    Complain

    Les violations sont enregistrées mais non bloquées.

    Avantages

    • simplicité ;
    • administration plus accessible ;
    • bonne intégration avec Debian et Ubuntu.

    6. Confinement des applications

    Principe

    Une application ne doit accéder qu'aux ressources nécessaires à son fonctionnement.

    Exemple : serveur Web

    Apache doit accéder :

    /var/www

    mais ne devrait pas accéder :

    /etc/shadow
/home

    Objectifs

    • limiter les dégâts ;
    • empêcher les mouvements latéraux ;
    • protéger les données sensibles.

    7. Le confinement dans une stratégie de défense en profondeur

    Une couche supplémentaire

    Même si :

    • l'authentification est correcte ;
    • les permissions sont bien configurées ;

    une application peut être vulnérable.

    Le rôle du confinement

    Réduire les conséquences :

    • d'une vulnérabilité ;
    • d'une erreur de configuration ;
    • d'une compromission.

    8. Synthèse

    À retenir

    • Les permissions Unix et les ACL reposent sur le modèle DAC.
    • DAC ne suffit pas toujours à protéger un système.
    • Le modèle MAC ajoute une couche de protection supplémentaire.
    • SELinux et AppArmor permettent de confiner les applications.
    • Le confinement contribue à la défense en profondeur en limitant l'impact d'une compromission.

    Préparation du TP

    Le TP portera sur :

    • l'observation des profils AppArmor ;
    • les modes Enforce et Complain ;
    • le confinement d'une application ;
    • une démonstration du fonctionnement de SELinux.

# Séance 8 – Vulnérabilités des plateformes logicielles

## Objectifs de la séance

À l'issue de cette séance, l'étudiant sera capable de :

- Comprendre comment une vulnérabilité applicative peut compromettre un système d'information ;
- Identifier les principales vulnérabilités présentes dans une application Web ;
- Comprendre les mécanismes d'exploitation de ces vulnérabilités ;
- Évaluer leurs impacts sur la confidentialité, l'intégrité et la disponibilité ;
- Identifier les mesures permettant de limiter les risques.

---

# 1. Introduction

## Des systèmes sécurisés mais des applications vulnérables

Au cours des séances précédentes, nous avons étudié :

- l'authentification ;
- le contrôle d'accès ;
- le confinement des applications ;
- la défense en profondeur.

Question :

> Un système correctement configuré est-il forcément sécurisé ?

La réponse est non.

Une application vulnérable peut permettre à un attaquant de contourner les mécanismes de protection du système.

---

## Plateforme logicielle

Une plateforme logicielle est composée de plusieurs éléments :

```text
Utilisateur
      ↓
Application Web
      ↓
Serveur Web
      ↓
Système d'exploitation
      ↓
Données
```

Une vulnérabilité dans l'application peut avoir des conséquences sur l'ensemble de la plateforme.

---

# 2. Surface d'attaque d'une application

## Définition

La surface d'attaque correspond à l'ensemble des points d'entrée exploitables.

---

## Exemples

- formulaires ;
- paramètres URL ;
- champs de recherche ;
- mécanismes d'authentification ;
- téléchargements de fichiers.

---

## Risques

Un attaquant peut :

- récupérer des informations ;
- contourner les contrôles ;
- exécuter des actions non autorisées.

---

# 3. Injection SQL

## Principe

Une entrée utilisateur est intégrée à une requête SQL sans validation suffisante.

---

## Exemple simplifié

```sql
SELECT * FROM users
WHERE login='admin'
AND password='motdepasse';
```

Une entrée malveillante peut modifier la requête.

---

## Conséquences

- accès non autorisé ;
- divulgation de données ;
- modification ou suppression de données.

---

## Impacts sur la sécurité

### Confidentialité

Accès aux informations sensibles.

### Intégrité

Modification des données.

### Disponibilité

Destruction ou corruption de données.

---

# 4. Inclusion de fichiers (File Inclusion)

## Principe

L'application charge un fichier dont le nom est fourni par l'utilisateur.

---

## Risques

Accès à :

```text
/etc/passwd
```

ou à d'autres fichiers sensibles.

---

## Conséquences

- divulgation d'informations ;
- découverte de la configuration du système ;
- préparation d'autres attaques.

---

# 5. Injection de commandes

## Principe

L'application exécute une commande système construite à partir d'une entrée utilisateur.

---

## Exemple conceptuel

L'application appelle :

```text
ping
```

sur une adresse fournie par l'utilisateur.

Une entrée malveillante peut permettre l'exécution de commandes supplémentaires.

---

## Conséquences

- accès au système ;
- consultation de fichiers ;
- modification de données ;
- compromission du serveur.

---

# 6. Impacts sur le système d'exploitation

## Atteinte à la confidentialité

Exemples :

- consultation de fichiers ;
- accès aux données utilisateurs ;
- récupération d'informations système.

---

## Atteinte à l'intégrité

Exemples :

- modification de fichiers ;
- altération de données ;
- changement de configuration.

---

## Atteinte à la disponibilité

Exemples :

- suppression de données ;
- interruption de service ;
- blocage de l'application.

---

# 7. Mesures de protection

## Validation des entrées

Contrôler les données fournies par les utilisateurs.

---

## Gestion des droits

Appliquer :

- permissions adaptées ;
- principe du moindre privilège.

---

## Confinement des applications

Limiter les ressources accessibles à l'application.

Exemple :

- AppArmor.

---

## Défense en profondeur

Combiner plusieurs mécanismes :

- authentification ;
- contrôle d'accès ;
- confinement ;
- mises à jour ;
- sécurité applicative.

---

# 8. Synthèse du module

## Les mécanismes étudiés

### Protection des données

- confidentialité ;
- intégrité ;
- disponibilité.

### Authentification

- PAM ;
- politiques de mots de passe.

### Contrôle d'accès

- permissions Unix ;
- ACL.

### Confinement

- AppArmor ;
- SELinux (présentation).

### Défense en profondeur

- moindre privilège ;
- réduction de la surface d'attaque.

---

## Message clé du module

Un système sécurisé repose sur plusieurs couches de protection complémentaires.

La sécurité ne dépend pas uniquement du système d'exploitation : les applications doivent également être conçues et configurées de manière sécurisée.

---

Le TP utilisera DVWA afin d'illustrer :

L'objectif sera d'analyser les impacts de ces vulnérabilités sur le système Linux et d'identifier les mécanismes permettant de limiter leurs conséquences.
